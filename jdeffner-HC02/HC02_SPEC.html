What the sheet demands, in one readable page

Two CUDA micro-benchmarks (compute-bound divergence, memory-bound bandwidth), a CPU baseline, a measurement pipeline that runs on Colab, and a report where every number is traceable to a committed CSV.

PR target: 2026S-HC deadline 2026-07-12 ~10 min presentation no personal info anywhere everything in English

Readable companion to HC02_SPEC.md (the normative text). Checklist entries in §7 carry the measured outcomes of the 2026-07-19 T4 run, clearly marked. See also: the code, explained · slide deck.

The one architectural idea

CSV is the interface between C++ and Python. Every binary prints CSV to stdout (header included); run_all.sh redirects it into results/. Every CSV starts with a provenance comment, # gpu=Tesla T4, cc=7.5, driver=13.0, toolkit=12.8, so the hardware behind any number is on record.

2Shared infrastructure (common.cuh)

  • CUDA_CHECK(call): abort with file/line on any CUDA error.
  • Timing: cudaEvent_t pair around the kernel only (no H2D/D2H in the timed region). Universal convention: 3 warmup launches, 10 timed launches, report the median.
  • print_device_info(): name, compute capability, SM count, max threads/SM, warp size, global memory, L2, memory clock and bus width. Attributes may be missing on newer toolkits → print n/a, never crash. Theoretical peak bandwidth = 2 × memClockHz × busWidthBytes when available. This replaces deviceQuery.
  • All binaries take CLI flags (hand-rolled parsing, no library) and document them via --help.

3Task 1 · SIMT and warp divergence (compute-bound)

Kernel design: one thread per element, minimal memory traffic: read one float, run k iterations of pure arithmetic, write one float. The inner load is an iterated FMA chain (x = fmaf(x, a, b), 2 FLOP per FMA) with per-iteration constant variation so the compiler cannot collapse the loop. Result written to global memory so nothing is dead-code-eliminated. float, fast math off, -O3.

3.2 Experiment A: throughput scaling over n

  • Sweep n ∈ {2^10, 2^12, …, 2^26}, fixed k = 1024, block 256.
  • Metric: GFLOP/s = 2·n·k / t_median.
  • CPU baseline: identical loop, (a) single thread, (b) OpenMP all cores.
  • The report must read off: at which n the GPU saturates, how many threads that is and per SM, versus the CPU which needs no such scale.
  • CSV: task1_scaling.csv · device,n,k,block,time_ms,gflops

3.3 Experiment B: controlled warp divergence

  • Divergence degree d ∈ {1, 2, 4, 8, 16, 32}: branch on threadIdx.x % d, so one warp executes d distinct paths. Fixed n = 2^24, k = 1024, block 256.
  • Critical design rule: every path performs the same work k, so any slowdown is pure serialization, not workload imbalance. Ideal expectation: slowdown ≈ d.
  • CPU comparison: same %d branching in the OpenMP baseline; expectation near-flat over d. Measure it, do not just claim it.
  • Secondary variant looplen: data-dependent trip count normalized to equal total work, showing divergence via loop length. Extra CSV column mode ∈ {branch, looplen}.
  • CSV: task1_divergence.csv · device,mode,d,n,k,time_ms,gflops,slowdown_vs_d1
Pitfall the spec insists on

If the d branch bodies are structurally identical and differ only in constants, nvcc merges them into one path with a constant lookup and the divergence disappears. Make each path operationally distinct (different arithmetic mixes, generated by template/macro so 32 cases stay maintainable). Verify the effect is real: measured d=32 slowdown must be roughly an order of magnitude or more vs d=1; otherwise inspect with --ptxas-options=-v and fix before proceeding.

Plots: task1_scaling.png (GFLOP/s vs n, log-x, three lines, saturation marked) and task1_divergence.png (relative throughput vs d, GPU + CPU + ideal 1/d reference).

4Task 2 · Memory access, latency hiding, bandwidth (memory-bound)

Kernel design: streaming kernel with minimal arithmetic intensity, b[j] = a[j] · c, n = 2^26 (256 MiB per array; check free memory, halve if needed), under three addressing patterns:

1 · coalesced

j = i (stride 1). Its measured result is the practical peak all percentages refer to.

2 · strided

j = (i mod chunk)·stride + i/chunk, stride ∈ {2 … 128}: a clean permutation, every element touched exactly once.

3 · random gather

b[i] = a[idx[i]]·c with a fixed-seed (42) uniform random permutation, built on the host, copied outside the timed region.

byte convention

8 B/element (4 read + 4 write). The gather’s 4 B/element index traffic is on top — footnote it in the report.

Correctness check: sum over b equals sum over a · c within tolerance, once per run.

4.2 Experiment C: pattern comparison

Sweep the three patterns (strided over its stride set), block 256, full grid. Reference: theoretical peak (from §2) and measured practical peak; express everything as % of practical peak.
CSV: task2_patterns.csv · pattern,stride,n,time_ms,gbps,pct_of_peak

4.3 Experiment D: latency hiding via occupancy

  • Coalesced kernel with a grid-stride loop so total work is identical across configurations.
  • Two knobs for resident warps: block-size sweep {32 … 1024}, and an occupancy throttle (fixed block 256, unused dynamic shared memory per block caps resident blocks/SM, target ~12.5% → 100%).
  • Theoretical occupancy from cudaOccupancyMaxActiveBlocksPerMultiprocessor in the CSV. Measured occupancy via Nsight Compute is usually blocked on Colab (counter permissions); note it and rely on the API value.
  • CSV: task2_occupancy.csv · knob,block,smem_bytes,active_warps_per_sm,occupancy,time_ms,gbps

Plots: task2_patterns.png (strided as line over stride, coalesced/random/peak as reference lines) and task2_occupancy.png (bandwidth vs occupancy, both knobs; expected shape: rising, then flat once enough warps hide the latency).

5Colab runner (run_colab.ipynb)

Cells in order: assert GPU present (nvidia-smi) → clone repo (URL/branch parameterized) → CMake build (raw-nvcc fallback cell) → run_all.sh → plot.py → zip results/ and download. run_all.sh must be idempotent and finish well under 10 minutes on a T4.

6Report and presentation

REPORT.md (2–4 pages equivalent): setup & methodology · Task 1 results with the required explanation (why a warp serializes divergent paths: single PC per warp pre-Volta, active-mask execution, Volta+ ITS still serializes divergent streams; why the same branch is nearly free on a CPU: independent control flow, branch prediction, speculation) · Task 2 results with the latency-hiding story (GPU: warp oversubscription; CPU: caches + prefetching) and the data-layout consequences (SoA, coalesced 32/128-B transactions, avoid gather, align/pad) · Limitations.

All numeric claims must come from the committed CSVs. Plots embedded with relative paths.

SLIDES.md: outline only, ~8 slides, 10 minutes: motivation, SIMT model, exp A, exp B, why serialization, memory patterns, occupancy, takeaways.

8Out of scope

  • No shared-memory tiling, streams, multi-GPU, half precision, CUB/Thrust.
  • No CI, no unit-test framework (the checksum validation suffices).
  • The PR itself (fork, branch, submit) is done manually by the author; the README makes the folder PR-ready.
