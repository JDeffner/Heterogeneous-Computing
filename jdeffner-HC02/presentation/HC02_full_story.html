Approach · 1/3
why this problem

Every program is limited by one of two things

Bottleneck 1 · compute

How fast can the chip calculate?

Aufgabe 1. Ruled by the SIMT execution model: warps of 32 threads on one instruction stream. Costs appear when threads are scarce or disagree.

Bottleneck 2 · memory

How fast can data reach the chip?

Aufgabe 2. Ruled by 32-byte memory transactions and latency hiding. Costs appear when access patterns waste transactions or too few warps are resident.

The sheet’s claim: the same chip moves by orders of magnitude along both axes depending purely on how it is asked to work. The job: prove it, with measurements that survive scrutiny.

why extremes

The compute probe does maximal math per byte, the memory probe minimal, so each experiment feels only its own axis. A mixed kernel would measure a blur of both.

why one knob

Each plot is then literally “metric vs. that knob”. Any change in the curve has exactly one possible cause. That is the entire argument for trusting the numbers.

Approach · 3/3
design decisions

Decisions I would defend in a review

countable work

FMA chains, fast math off

One FMA = 2 FLOP, k per thread, so GFLOP/s is arithmetic, not estimation. Fast math would let the compiler change how many operations actually run.

compiler defense

32 operationally distinct paths

Distinct constants, distinct instruction order, __noinline__. Otherwise nvcc merges the branches and Experiment B silently measures nothing. Verified: 19.9× > the ~8× gate.

equal work rule

Divergence isolated from imbalance

Every path does identical work; the loop-length variant is even normalized to identical total work per warp. Any slowdown is serialization, nothing else.

built-in referee

Checksums, or it did not happen

All memory patterns are permutations, so sum(b) must equal sum(a)·c. Every configuration is validated; a wrong result aborts the run. All runs: relative error 0.0.

Plus: query everything at runtime, hardcode nothing (spec rule)CODE_EXPLAINED.html

Setup · 2/2
measurement rules

Rules that make the numbers boring, i.e. right

what is timed

The kernel, nothing else

cudaEvent markers on the GPU’s own timeline, placed directly around the kernel. Data transfers happen outside the window. Launches are asynchronous, so a CPU clock would time the wrong thing.

how often

3 warmups, median of 10

First launches pay one-time costs (code upload, clock ramp); the median absorbs cloud hiccups. One convention for GPU and CPU, so every number is comparable.

what can go wrong

Dead code and folded loops

Every result is written back to memory; chains are serially dependent. A benchmark the compiler can delete or precompute measures an empty kernel.

what cannot be controlled

Cloud clocks

No ECC/boost control on Colab. Visible in the data as level shifts between runs; handled by reporting plateau values and naming the outliers instead of hiding them.

Convention stated once in REPORT.md §1, used everywheresrc/common.cuh
Experiment A · design
compute axis · knob: n

When does a GPU earn its keep at all?

  • Hypothesis: GPU throughput collapses below mass parallelism (latency hiding needs many resident warps; 40 SMs need filling), while the CPU is flat from the first data point.
  • Procedure: uniform kernel (zero divergence), sweep n = 4 → 67 million in ×4 steps, k = 1024, block 256. CPU: identical loop, 1 thread and all cores.
  • Deliberately extended downward (n = 4!): with tiny n, the GPU’s fixed ~10 µs launch overhead dominates and the GPU must fall below the CPU. A honest scaling plot should show both ends.
what to look for

Three regimes, one curve

1 — overhead-dominated: GPU under the CPU line.
2 — ramp: SMs filling, latency still visible.
3 — plateau: every SM saturated, real throughput.

The CPU line has no regimes. That asymmetry is the result.

Experiment B · design
compute axis · knob: d

Charging the disagreement tax, cleanly

  • Hypothesis: a warp executes d distinct paths serially under the active mask → slowdown ≈ d. The CPU, with independent control flow per core and a perfectly predictable i mod d pattern, should be flat.
  • Procedure: branch on threadIdx.x % d, d ∈ {1 … 32}, n = 2^24, k = 1024. Every path: identical work.
  • Secondary probe: same total work but data-dependent loop lengths (looplen): divergence via trip count, expected to saturate near 2× (warp time = slowest lane, max k_r ≈ 2k).
the trap this design dodges

The compiler wants to merge your branches

Structurally identical bodies with different constants get folded into one path + lookup, and the experiment evaporates. Countermeasures: per-path constants and instruction order, __noinline__, and a mandatory verification gate (≥ ~8× at d = 32, else inspect the PTX).

Design rule: equal work per path, or the result is ambiguoustask1_divergence.csv

Experiment C · design
memory axis · knob: pattern

Same bytes, three visiting orders

  • Hypothesis: DRAM serves fixed 32-byte sectors. A warp of neighbors is a handful of full sectors; strided or random lanes burn one sector per element, so bandwidth should collapse by up to ~8× from waste alone, more with worse locality.
  • Procedure: b[j] = a[j]·c over 67 M floats. Coalesced (j = i), strided (permutation formula, stride 2 … 128), random gather (fixed-seed shuffled index array, built on the host, copied outside the timed window).
  • Accounting: 8 B/element (4 read + 4 write). The gather’s extra 4 B/element of index traffic is footnoted, not hidden.
why a permutation formula

Every element exactly once

Naive striding revisits some elements and skips others, so the three patterns would not move the same data. The matrix-transpose-style mapping keeps thread neighbors stride apart while touching each element once, and the checksum proves it.

Experiment D · design
memory axis · knob: occupancy

Throttling warps without touching the code

  • Hypothesis: bandwidth rises with resident warps until memory latency is fully hidden, then plateaus at the DRAM limit. Occupancy is fuel, not a score.
  • Trick 1, fairness: grid-stride loop, so 8 or 32 resident warps process the identical 67 M elements.
  • Trick 2, the throttle: each block reserves unused dynamic shared memory; reserving smem/m per block caps residency at m blocks/SM. Achieved residency is always re-queried from the occupancy API, never assumed.
  • Two series: block-size sweep 32 … 1024, and the smem throttle at block 256 and block 128, the latter reaching the 12.5% floor (4 warps/SM) that block 256 cannot express.
why “theoretical” occupancy

The profiler is locked out

Nsight Compute needs counter permissions Colab does not grant. The runtime API’s occupancy value is used instead and this is named openly in the limitations, rather than presenting profiled numbers that do not exist.

Honesty section
limitations

What these numbers do not claim

  • One GPU model. Everything is queried at runtime and runs on any cc ≥ 7.0 device, but the numbers describe a T4, nothing else.
  • No clock control. Colab’s boost clocks drift between runs: the 6,446 GFLOP/s outlier at n = 2^26, and level shifts of ~20% between sweeps. Medians tame noise within a configuration, not drift across them; plateau values are reported for that reason.
  • A weak CPU baseline, by construction. 2 cores, and the dependent scalar FMA chain cannot be vectorized, so the 1,600× headline compares this workload, not architectural peaks. Stated in the report before anyone has to ask.
  • Theoretical occupancy, not profiled. Colab denies profiler counters; the API value is used and labeled.
  • The extended low-n sweep measures kernel time only. Including PCIe transfers would make the GPU look worse still for small jobs; excluded by design (spec), noted for context.

Takeaways
what the data paid for

Four rules, four price tags

rule 1

Saturate first.

Below ~1 M threads the T4 never showed rated speed; below ~16 threads it lost to a 2-core CPU outright. Small problems belong on the CPU.

rule 2

Keep warps convergent.

32 paths per warp cost 19.9×. If branching is unavoidable, branch at warp granularity, not per lane.

rule 3

Layout is destiny.

243 vs 6.4 GB/s for identical work. Structure-of-arrays, neighbors reading neighbors, no gathers you can avoid.

rule 4

Occupancy is a means.

The plateau began at 50%. Enough resident warps to hide latency, then stop optimizing for the number.

Same silicon in every measurement. Factors of 21, 38, 1,600 appeared and vanished based only on how the work was presented, which is what the sheet asked to be demonstrated.
