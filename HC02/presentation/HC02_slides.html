Motivation

Two questions, one micro-benchmark each

Task 1 · compute-bound

What does control flow cost?

One float in, 1024 fused multiply-adds, one float out. First scale the thread count until the GPU saturates, then force a warp to execute d = 1 … 32 distinct paths and watch the price of divergence.

Task 2 · memory-bound

What does data layout cost?

A streaming kernel, b[j] = a[j]·c, under three address patterns: coalesced, strided, random gather. Then the same kernel with the number of resident warps throttled, to see latency hiding at work.

Same silicon in every run. The only thing that changes is how we ask it to work, and the answer spans two orders of magnitude.

Methodology

Measure honestly, or not at all

Timing

Kernel only, median of 10

cudaEvent pairs around the kernel, no transfers in the timed window. 3 warmup launches absorb one-time costs; the median of 10 timed launches shrugs off cloud noise.

Workload

Countable FLOPs

Pure FMA chains, 2 FLOP per instruction, fast math off. Every result is written back, so the compiler can delete nothing and the GFLOP/s bookkeeping is exact.

Portability

Nothing hardcoded

SM count, peak bandwidth, occupancy, free memory: all queried at runtime. The same binaries run on any GPU of compute capability ≥ 7.0.

Integrity

Validated & reproducible

Every memory kernel is checksummed against a permutation-invariant reference. Fixed seeds, CSVs with hardware provenance, plots regenerated by one script.

The model

A warp is 32 threads on one instruction stream

  • One program counter per warp (pre-Volta). A branch cannot fork it, so divergent targets execute sequentially under an active mask.
  • Volta+ gives each thread its own PC, but a scheduler still issues one instruction per cycle per lane group. Divergent instruction streams stay serialized.
  • Hypothesis for the experiment: d distinct paths per warp should cost a factor of d.

Explanation

Serialization vs. speculation

GPU · why it pays factor d

One instruction stream per warp

  • Pre-Volta: a single program counter for 32 lanes. Divergent targets run one after another under the active mask.
  • With d paths, each pass keeps only 32/d lanes busy: d× the time for the same work.
  • Volta+ independent thread scheduling changes bookkeeping, not the economics: divergent streams still issue serially.
CPU · why the same branch is ~free

Every core is its own machine

  • Independent control flow per core: no lane group that has to wait through foreign paths.
  • The pattern i mod d is perfectly periodic: the branch predictor learns it, prediction is near 100%.
  • Speculative, out-of-order execution hides what little remains: the measured curve stays flat over d.

Same source-level if, two opposite hardware answers. This asymmetry, not raw FLOPs, is the first thing that decides what belongs on which device.

Task 2 · Experiment D · latency hiding

Enough warps, and DRAM becomes the wall

  • Latency hiding is oversubscription: on a memory stall the scheduler switches to the next ready warp, so more resident warps mean fewer idle cycles.
  • Beyond the knee the DRAM interface is saturated; extra occupancy buys nothing.
  • Occupancy values are the runtime API's theoretical ones: Nsight Compute counters are blocked on Colab (noted as a limitation).

Takeaways

Four rules the data pays for

Saturate first.

A GPU below its saturation point is an expensive space heater. If the problem cannot flood it with threads, it belongs on the CPU.

Keep warps convergent.

d paths per warp cost d×. If branching is unavoidable, branch at warp granularity, not per lane.

Layout is destiny.

Structure-of-arrays, coalesced 32/128-byte transactions, no gathers you can avoid. The same bytes move up to an order of magnitude slower when scattered.

Occupancy is a means.

Enough resident warps to hide latency, then stop optimizing for it. The plateau is the DRAM's opinion, not yours.
